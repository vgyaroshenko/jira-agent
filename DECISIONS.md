# Лог рішень — Jira QA Agent

Кожен запис пояснює ЧОМУ прийнято те чи інше рішення.
Захищає від "чому так зроблено?" і від повторного наступання на ті самі граблі.

Формат: `## [дата] Назва рішення`

---

## [2026-06-12] fetch повертає коментарі задачі

**Рішення:** Команда `fetch` тепер підтягує коментарі через окремий виклик `/rest/api/3/issue/{key}/comment` і виводить їх після AC.

**Чому:** Без коментарів контекст задачі неповний — уточнення, відповіді Supriya, action items часто живуть саме в коментарях, а не в описі.

---

## [2026-06-01] AskUserQuestion замість інтерактивного терміналу

**Рішення:** Для вибору мови і типу задачі використовується інструмент `AskUserQuestion` (через Claude Code), а не `questionary` в терміналі.

**Чому:** `questionary` вимагає TTY і не працює коли агент запускає команди через `Bash`. Спершу спробували нумеровані варіанти у stdout, але це некрасиво і ненадійно. `AskUserQuestion` дає нативний UI прямо в чаті.

**Де:** `jira_client.py` не змінювався — вибір відбувається до виклику команди, `--lang` і `--type` передаються явно.

---

## [2026-05-28] Команда update для редагування існуючих задач

**Рішення:** Додано `python main.py update ISSUE-KEY [--title "..."] [--lang XX]` з описом зі stdin.

**Чому:** Виникла потреба виправляти заголовок і опис після створення задачі. Замість окремих команд для title і description — одна гнучка, де будь-який параметр опційний.

**Важливо:** якщо передається тільки `--title` — `--lang` не потрібен. Формат опису — ті самі маркери `##DESC##`, `##SECTION##`, `##AC##`.

---

## [2026-05-20] Структуровані маркери ##...## для описів задач і багів

**Рішення:** Опис передається зі stdin у форматі з маркерами (`##DESC##`, `##SECTION##`, `##AC##` для задач; `##ENV##`, `##STEPS##` тощо для багів). `jira_client.py` парсить маркери і конвертує в ADF.

**Чому:** Jira API приймає тільки ADF (Atlassian Document Format). Простий текст не форматується. Маркери дають передбачуваний маппінг: секція → heading, список → bulletList, кроки → orderedList.
