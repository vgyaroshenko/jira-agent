# Jira QA Agent

AI-помічник для QA-інженера. Працює через Claude Code — без окремого API-ключа Anthropic.

## Налаштування

```bash
cd /Users/volod/dev/claude/jira-agent
source .venv/bin/activate
```

Налаштування Jira — у файлі `.env`.

---

## Команди

| Команда | Що робить |
|---|---|
| `python main.py fetch GN-1808` | Отримати дані задачі з Jira |
| `python main.py comment GN-1808 [--mention ID]` | Додати коментар (текст зі stdin) |
| `python main.py new-bug --title "..." --lang UA [--related GN-123]` | Створити баг (опис зі stdin) |
| `python main.py sprint GN` | Вивести задачі активного спринту |

## Генерація тест-кейсів

1. Отримай дані задачі:
   ```bash
   python main.py fetch GN-1808
   ```
2. Згенеруй тест-кейси за форматом нижче
3. Запости:
   ```bash
   echo "тест-кейси" | python main.py comment GN-1808
   ```
   або з heredoc для багаторядкового тексту:
   ```bash
   python main.py comment GN-1808 << 'EOF'
   тест-кейси...
   EOF
   ```

### Формат тест-кейсів

Мова тест-кейсів визначається за заголовком задачі: і/ї/є/ґ → UA, ё/ы/ъ/э → RU, інше → EN.

Генеруй тест-кейси у вигляді markdown-таблиці з чотирма колонками:

| Колонка | Що містить |
|---|---|
| **Case** | Назва кейса — що саме перевіряємо (TC-001 — Назва) |
| **Description** | Передумови або контекст. Якщо немає — залишай порожнім |
| **Steps** | Нумеровані кроки, розділені символом `\n`. Після кроків — `ОР:` з очікуваним результатом |
| **Priority** | `High` — критичний функціонал, `Medium` — основні сценарії, `Low` — граничні та негативні |

**Приклад виводу:**

```
🤖 Автоматично згенеровані тест-кейси

| Case | Description | Steps | Priority |
|---|---|---|---|
| TC-001 — Успішна авторизація | Користувач зареєстрований в системі | 1. Відкрити сторінку входу\n2. Ввести коректний логін і пароль\n3. Натиснути «Увійти»\nОР: Відбувається перехід на головну сторінку | High |
| TC-002 — Авторизація з невірним паролем | | 1. Відкрити сторінку входу\n2. Ввести коректний логін та невірний пароль\n3. Натиснути «Увійти»\nОР: З'являється повідомлення «Невірний пароль», сторінка не оновлюється | Medium |
| TC-003 — Порожні поля | | 1. Відкрити сторінку входу\n2. Не заповнювати жодне поле\n3. Натиснути «Увійти»\nОР: Поля підсвічуються як обов'язкові, форма не відправляється | Low |
```

Символ `\n` у колонці Steps — це буквальні два символи `\` і `n`, не перенос рядка. Код конвертує їх у перенос самостійно при відправці в Jira.

Покривай: позитивні сценарії, негативні, граничні випадки, UI/UX якщо релевантно.

## Створення баг-репорту

Коли користувач описує баг у вільній формі — дотримуйся цього порядку строго:

1. Проаналізуй опис — чи зрозуміло що саме зламалося і за яких умов
2. Якщо критично не вистачає інформації — задай **одне** уточнювальне питання
3. Після отримання достатньо інформації — **обов'язково запитай мову** перед генерацією:
   > На якій мові створити баг-репорт? **UA**, **RU** або **EN**?
4. Дочекайся відповіді — не генеруй баг-репорт до її отримання
5. **Не питай** про кроки відтворення та очікуваний результат — виведи їх сам
6. Згенеруй баг-репорт у форматі нижче і запости через `new-bug`

### Шаблон баг-репорту

Маркери `##...##` — обов'язкові, назви полів — мовою задачі:

```
##ENV##
Середовище: PROD / STAGING / DEV / Не вказано
Браузер: значення або Не вказано
Розділ: значення або Не вказано
Сторінка: значення або Не вказано

##DESC##
Розгорнутий опис проблеми — 2-4 речення.

##STEPS##
1. Крок перший
2. Крок другий
3. Крок третій

##EXPECTED##
- Очікувана поведінка

##ACTUAL##
- Фактична поведінка

##ADDITIONAL##
Додаткова інформація або Не вказано
```

### Команда для створення бага

```bash
python main.py new-bug --title "Короткий заголовок до 80 символів" --lang UA << 'EOF'
##ENV##
Середовище: PROD
...
EOF
```

Прапор `--related GN-123` — якщо баг пов'язаний з конкретною задачею (баг створиться в тому ж проекті).

## Створення задачі

Коли користувач просить створити задачу — дотримуйся цього порядку кроків строго:

1. Якщо користувач не вказав деталі — запитай що потрібно реалізувати
2. Отримавши деталі — **обов'язково постав два питання** перед тим як щось генерувати:
   > Що створити: **Story** або **Task**?
   > На якій мові створити задачу? **UA**, **RU** або **EN**?
3. Дочекайся відповіді на обидва питання — не генеруй задачу до їх отримання
4. Отримавши відповіді — згенеруй задачу у форматі нижче і запости через `new-task` з відповідним `--type`

### Правила якості

**DESC** — не переписуй те що сказав користувач. Напиши чіткий вступний абзац: що це за задача і яку бізнес-проблему вирішує. 1-2 речення.

**SECTION** — кожна пронумерована вимога або логічний блок — окрема секція. Всередині:
- Кожен пункт — окремий буллет
- Якщо є конкретний текст для відображення, приклад або уточнення — включи як окремий буллет
- Якщо є підпункти (перший продукт → знижка, другий → повна ціна) — включи їх як окремі буллети

**AC** — генеруй самостійно, навіть якщо користувач їх не надав. Для кожного AC:
- Перший рядок: назва сценарію
- Другий рядок: умова (що відбувається / що активне)
- Третій рядок: `✅ Очікуваний результат: ...` — конкретний, вимірюваний результат
- Покривай всі ключові сценарії із задачі

### Шаблон

```
##DESC##
Вступний опис — що потрібно реалізувати і яку проблему це вирішує.

##SECTION## 1. Назва першої вимоги
- Умова або контекст
- Конкретна поведінка яку потрібно реалізувати
- Якщо є специфічний текст для відображення — включи його як окремий буллет
- Якщо є приклад — включи як окремий буллет

##SECTION## 2. Назва другої вимоги
- Основна поведінка
- Підпункт або приклад якщо є

##AC##
AC1. Назва сценарію\nУмова: що активне або що відбувається.\n✅ Очікуваний результат: конкретний результат.
AC2. Назва сценарію\nУмова.\n✅ Очікуваний результат: конкретний результат.
```

### Повний приклад

Запит користувача: *"Реферальна акція і welcome bonus не можуть використовуватись одночасно. Треба показувати попередження на сторінках акцій і при додаванні товарів перша знижка застосовується, друга — ні."*

Очікуваний вивід:

```
##DESC##
Необхідно реалізувати обмеження, при якому реферальна акція (-50) і welcome bonus не можуть використовуватися одночасно в одному замовленні.

##SECTION## 1. Інформування на сторінках акцій
- Якщо у користувача одночасно доступні реферальна акція –50 і welcome bonus — на сторінках обох акцій виводити попереджувальний текст у верхній частині опису
- Текст для відображення (українська мова):\nРеферальна знижка -50 не може бути використана одночасно з welcome bonus в одному замовленні.

##SECTION## 2. Логіка застосування знижки
- Якщо користувач додає в кошик товари з обох акцій — застосовується лише перша додана знижка
- Товари, додані після — йдуть за повною вартістю
- Приклад:\nПерший доданий товар з реферальної акції → зі знижкою.\nДругий товар з welcome bonus → без знижки (повна ціна).

##AC##
AC1. Перевірка доступності акцій\nУ користувача активні обидві акції: реферальна і welcome bonus.\n✅ Очікуваний результат: на сторінках обох акцій відображається попереджувальний текст.
AC2. Відображення попередження\nТекст знаходиться у верхній частині опису акції.\n✅ Очікуваний результат: користувач бачить повідомлення про неможливість спільного використання.
AC3. Застосування знижки\nТовари з обох акцій додані в один кошик.\n✅ Очікуваний результат: застосовується лише перша знижка, друга ігнорується, знижки не підсумовуються.
AC4. Коректне відображення цін\nДругий товар доданий з іншої акції.\n✅ Очікуваний результат: ціна другого товару відображається повною, без застосування знижки.
```

### Команда для створення задачі

```bash
python main.py new-task --title "Заголовок задачі" --lang RU --project GN << 'EOF'
##DESC##
...

##SECTION## 1. Назва
- ...

##AC##
AC1. ...
EOF
```

Опційно `--type Task` якщо потрібен тип Task замість Story (за замовчуванням Story).

## Перевірка якості задачі

1. Отримай дані:
   ```bash
   python main.py fetch GN-1808
   ```
2. Перевір наявність обов'язкових полів:

**Для звичайних задач:**
- Конкретний заголовок (не "Оновити сторінку", а суть)
- Непорожній опис
- Зрозумілий очікуваний результат

**Для багів (TYPE: Bug):**
- Конкретний заголовок
- Кроки відтворення
- Очікуваний результат (ОР, Expected result)
- Фактичний результат (АР, Actual result)

3. Якщо недостатньо — залиш коментар з тегом репортера:
   ```bash
   python main.py comment GN-1808 --mention <REPORTER_ID> << 'EOF'
   🤖 @репортер, будь ласка, доповни задачу:
   - ...
   EOF
   ```
   `REPORTER_ID` є у виводі `fetch` (поле `REPORTER_ID`).

## Перевірка спринту

```bash
python main.py sprint GN
```

Виводить список задач з позначками:
- `✅TC` — тест-кейси вже є
- `❌TC` — тест-кейсів немає
- `✅QC` — коментар якості вже є
- `❌QC` — коментаря немає

Для кожної задачі з `❌TC`: зроби `fetch` → перевір якість → якщо OK → згенеруй тест-кейси → `comment`.
